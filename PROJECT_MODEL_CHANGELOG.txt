Описание проекта
- Этот файл служит единым источником информации для моделей и систем, взаимодействующих с проектом. Здесь фиксируются изменения, внесённые любыми моделями, а также текущее состояние проекта, чтобы новые участники или агенты могли быстро вникнуть в суть и контекст.

Структура файла
- Каждая запись об изменении добавляется как самостоятельный блок и отделяется горизонтальной строкой условий, чтобы избежать перемешивания записей.
- Формат записей предельно прост: каждая запись содержит поля, отделённые переносами строк.

Формат записи изменений (одна запись):
Дата: ГГГГ-ММ-ДД ЧЧ:мм:сс
Модель: <id модели>
Тип изменений: <feature|fix|update|refactor|docs>
Описание: <краткое описание изменений>
Затронутые компоненты: [<компонент1>, <компонент2>, ...]
Идентификатор изменений: <external_id или ссылка>
Версия проекта: <semver, например 0.1.0>
Примечания: <любые дополнительные замечания>
------------------------------------------------------------

Шаблон заполнения (пример):
Дата: 2026-04-29 12:34:56
Модель: model-alpha
Тип изменений: feature
Описание: Добавлена интеграция с внешним API A
Затронутые компоненты: [backend/api, frontend/worker]
Идентификатор изменений: #1234
Версия проекта: 0.1.0
Примечания: Включена базовая авторизация для нового API.
------------------------------------------------------------

Инструкция для моделей
- Каждая модель обязана добавлять новую запись изменений в этот файл после внесения изменений.
- Не редактируйте уже существующие записи; новые изменения должны всегда добавляться в конец файла.
- При необходимости обновлять общую информацию о проекте (например версия проекта, описание), используйте соответствующие поля в новой записи или обновляйте текстовую секцию, если это предусмотрено вашей архитектурой.
- В случае одновременных записей нескольких моделей применяйте совместную координацию: создайте локальный временный механизм блокировки (например, файл LOCK) перед записью и удаляйте его после завершения записи.

Дополнительная информация (рекомендации)
- Включайте в поле Затронутые компоненты все релевантные части проекта, чтобы можно было быстро понять влияние изменений.
- для идентификаторов изменений используйте как внешний номер задачи/тикета, так и внутренний номер; это ускоряет поиск по истории.
- по возможности сохраняйте минимально достаточное описание: этого достаточно для понимания суть изменений без чтения кода.

Пример статуса проекта (опционально, обновляйте по мере необходимости):
- Версия проекта: 0.1.0
- Ключевые зависимости: API A, API B
- Текущие риски: [описать]
- Следующие шаги: [описать]

Примечание: файл предназначен для моделей и агентов, которые взаимодействуют с проектом. Любые вспомогательные данные должны документироваться здесь же.
